## 任务介绍

在真实科研与模型研发过程中，实验优化往往不是一次性完成的，而是围绕“提出配置—运行试验—读取反馈—决定下一步”的闭环持续展开。研究者通常无法提前获得完整监督，而只能看到有限的中间反馈，例如验证集指标、训练耗时、资源占用、失败日志等。如何在有限预算内利用这些稀疏反馈持续修正实验策略，是自动化科研与智能体研究中的核心问题之一。

本赛题面向**Agent 长程推理与自动化实验控制**，聚焦有限预算、稀疏反馈、禁止并行约束下的自动化实验智能体构建。参赛者需要设计一个能够在多轮交互中持续优化实验决策的 agent：读取历史试验结果，归纳有效经验，决定下一轮模型、超参数或训练策略，并在预算耗尽前输出尽可能优的预测结果。

为保证评测标准化与可复现性，赛题采用金融场景下的**图学习任务**作为测试任务，并设置两个具体子任务：

（1）**产品分类任务**：以图节点分类的形式进行评测；

（2）**产品推荐任务**：在用户-产品的二部图上的图链接预测问题。

两个子任务都面向稀疏监督场景，即测试节点/用户可利用的邻域或历史信息较少。比赛重点不在于手工调参或单一模型结构本身，而在于 Agent 是否具备以下能力：在多轮实验中维护有效的实验记忆；从稀疏反馈中提炼规律并更新后续策略；在预算约束下进行合理的探索、利用与停止决策；从不同数据集的实验过程中总结可迁移经验用于新的图任务。

为便于理解，可以将本赛题看作一个受控的金融模型研发沙盒。以新用户金融产品推荐为例，Agent 首先需要根据已有用户行为、产品特征和历史交互结果生成一个初始实验方案；随后根据有限反馈判断当前方案的不足，例如部分新用户预测不稳定、某类产品召回不足、模型训练成本过高，或验证结果与线上反馈不一致；接着，Agent 需要自主决定下一轮实验方向，例如调整特征使用方式、改变模型结构、修改训练策略、重新分配实验预算，或在收益有限时提前停止。经过多轮迭代后，Agent 需要在预算耗尽前选择当前最优方案并生成最终提交结果。

因此，本题是一个“**受控机器学习实验环境中的长程决策问题**”，也是一个兼具研究性与工程性的综合挑战题。赛题不仅关注最终预测结果的准确性，也关注 Agent 系统在有限预算下如何组织多轮实验、利用反馈信息、形成优化策略并逐步改进预测器。

## 二、任务目标

本赛题围绕“**稀疏反馈下的自动化实验控制**”展开，参赛者需要面向给定的图学习任务，设计一个能够自主开展多轮实验的智能体系统。该系统需要在资源受限条件下，自动完成“读取历史实验结果—分析反馈信号—决定下一轮实验配置—持续迭代优化—输出最终预测结果”的完整流程。

本赛题包含两个具体子任务：

- **产品分类任务**：参赛者需要根据给定图数据（含产品节点特征与节点间连接关系），对测试节点进行类别预测；
- **产品推荐任务**：参赛者需要根据给定用户产品交互序列、用户特征和产品特征，为每个测试用户预测一个按置信度排序的产品列表。

两个子任务都面向典型的**稀疏监督场景**，在产品分类任务中，参赛者需要对**连接关系较为稀疏的节点**进行分类；在产品推荐任务中，参赛者需要对**仅具有少量交互历史的新用户**进行推荐。因而参赛方法不能依赖密集的监督信号，而需要在有限反馈、有限预算和多轮实验过程中，逐步归纳有效经验并优化决策策略。

参赛者需要构建一个自动化实验 Agent系统。对于每个待求解的任务与数据集，Agent 需要自主决定尝试的预测器模型结构、超参数配置、训练策略或停止时机，并在有限预算内不断改进结果，最终得到预测器来完成预测任务。在比赛提交阶段，参赛者需要按照赛题要求提交预测结果文件，平台将根据预测结果进行自动评测。进入复审阶段的队伍还需提交完整代码，以证明其方法能够在规定环境中稳定复现。

本赛题鼓励参赛者结合图学习、自动化搜索、实验记忆、规划与反思、多 Agent 编排、RAG，以及主办方允许的 Qwen 系列模型 API 等技术来解决问题。考虑到复现阶段对代码体积和运行环境的要求，参赛者应优先设计轻量、稳定、可审计的预测器模型与 Agent 系统，而不是依赖难以复现的本地大模型。

为帮助参赛者理解任务形式，赛题将提供公开评测数据、统一输入输出格式、提交模板及相关说明文档。参赛者需要基于这些材料，构建一个能够在未知图任务上持续优化实验决策并输出高质量预测结果的自动化实验系统。

## 三、任务数据

本赛题采用 A/B 榜机制，A 榜将提供初赛测试集，用于模型调试、提交格式校验与阶段筛选；B 榜将提供复赛测试集，用于代码提交阶段的自动评测排名。

A 榜、B 榜都各包含 1 个产品分类数据集以及1 个产品推荐数据集。A/B 榜在任务形式、输入输出接口、运行约束和评测指标上保持一致，区别主要在于数据划分、测试目标以及是否用于最终排名。

## 四、数据集构成

### （一） 产品分类任务数据

产品分类任务以 `.npz` 文件形式提供，核心包含图结构、节点特征、标签及训练/测试节点索引。训练节点与验证节点合并提供，不再单独给出官方验证集索引；测试节点标签隐藏，需要选手预测。所有节点编号均为从 `0` 开始的整数编号。

数据读取后的逻辑可概括为：

- 图结构：邻接矩阵 CSR 表示；
- 节点特征：特征矩阵 CSR 表示；
- 标签：训练节点标签公开，测试节点标签隐藏；
- 训练/测试划分：通过 `train_idx` 和 `test_idx` 给出。

数据集示例说明：

示意数据集包含 4 个节点（节点编号为 0,1,2,3）和 4 条边，类别数为 2，节点特征维度为 4。其 .npz 文件中各字段可示意如下：

```json
{

   "adj_data": [1.0, 1.0, 1.0, 1.0],

   "adj_indices": [1, 2, 3, 0],

   "adj_indptr": [0, 1, 2, 3, 4],

   "adj_shape": [4, 4],

   "attr_data": [1.0, 0.5, 1.0, 0.2, 1.0, 1.0, 1.0, 1.0],

   "attr_indices": [0, 2, 1, 3, 0, 1, 2, 3],

   "attr_indptr": [0, 2, 4, 6, 8],

   "attr_shape": [4, 4],

   "labels": [0, 1, -1, -1],

   "train_idx": [0, 1],

   "test_idx": [2, 3]

}
```

|  **变量名**  |                           **含义**                           |
| :----------: | :----------------------------------------------------------: |
|   adj_data   |          图邻接矩阵 CSR 的 data 数组，表示非零边权           |
| adj_indices  |   图邻接矩阵 CSR 的 indices 数组，表示边指向的目标节点编号   |
|  adj_indptr  | 图邻接矩阵 CSR 的 indptr 数组，用于划分每个节点对应的邻接边范围 |
|  adj_shape   |            邻接矩阵形状，格式为 (节点数, 节点数)             |
|  attr_data   |        节点特征矩阵 CSR 的 data 数组，表示非零特征值         |
| attr_indices | 节点特征矩阵 CSR 的 indices 数组，表示非零特征所在的特征列编号 |
| attr_indptr  | 节点特征矩阵 CSR 的 indptr 数组，用于划分每个节点对应的特征范围 |
|  attr_shape  |           特征矩阵形状，格式为 (节点数, 特征维度)            |
|    labels    | 节点标签数组；公开训练节点位置为真实类别，测试节点位置为 -1  |
|  train_idx   |                可用于训练/验证的节点编号数组                 |
|   test_idx   |          测试节点编号数组，需对这些节点输出预测结果          |

- 节点 0 -> 1
- 节点 1 -> 2
- 节点 2 -> 3
- 节点 3 -> 0

### （三） 校验规则及释义

**产品分类任务**校验规则如下：

1. 节点编号规则：所有节点编号均为从 `0` 开始的整数编号。
2. 标签规则：`labels` 中，训练节点位置为公开标签；测试节点位置统一为 `-1`，不代表真实类别。
3. 训练集规则：`train_idx` 为可用于训练/验证的节点集合，选手需自行制定内部验证策略。
4. 测试集规则：`test_idx` 为待预测节点集合。选手需对这些节点输出类别编号，并严格按照提交模板中的顺序填写。
5. 类别编号规则：提交的 `label` 必须为合法整数类别编号。A 榜类别范围为 `0` 到 `9`。B 榜类别范围以 B 榜数据说明文件为准。

**产品推荐任务**校验规则如下：

1. 用户编号规则：提交文件中的 `uid` 必须与官方 `test.csv`保持一致，不得删除、增加或重排用户行。
2. 产品编号规则：`prediction` 中的 item id 必须来自对应数据包 `item.csv` 的 `iid` 候选集合。
3. 排序规则：`prediction` 中的 item id 需按置信度从高到低排序，总计10个，使用英文逗号分隔。
4. 去重规则：同一用户的推荐列表中不应出现重复 item id；若出现重复、非法 id 或缺失预测，平台将按评测脚本规则处理，可能导致该用户得分降低。

## 五、任务规则及提交说明

### （一） 提交文件

#### **代码提交**

选手在 A 榜的线上评测阶段，每次提交仅需提交预测结果文件，将产品分类任务的预测结果（`A1.csv`）和产品推荐任务预测结果`A2.csv`压缩到一个压缩包中，命名为`prediction.zip`。

产品分类任务提交文件命名为`A1.csv`，必须包含表头，且仅允许包含两列，例如：

test_idx,label

18,3

19,7

其中：· `test_idx`：测试节点编号，必须与官方提供数据集保持一致。· `label`：选手预测的类别编号，必须为合法整数类别编号。

产品推荐任务提交文件命名为`A2.csv`，必须包含表头，且仅允许包含两列，例如：

uid,prediction

u000001,"i000001,i000002,i000003,i000004,i000005,……"

其中：· `uid`：测试用户编号，必须与官方测试集保持一致。· `prediction`：按置信度从高到低排序的 item id 列表（长度为10），使用英文逗号分隔。默认候选 item 集合为 `item.csv` 中全部 `iid`。

### （二） 评分规则和脚本

#### **代码提交阶段自动评测**

平台将在统一环境下对选手提交结果进行自动评测。产品分类任务和产品推荐任务分别计算得分，最终榜单分数由二者加权得出。

产品分类任务采用 Accuracy 作为评测指标，得分为测试节点上的平均预测准确率。

产品推荐任务采用 NDCG@K 作为评测指标，默认 K=10。即对每个测试用户，检查隐藏目标 item是否出现在预测列表中以及排位情况；再对全部测试用户取平均。最终榜单分数为节点分类任务分数Score_cls和推荐任务分数Score_ndcg的平均：

Score_final = 0.5 * Score_cls + 0.5 * Score_ndcg

#### **API 与外部资源规则：**

1. 允许调用出题方指定的大模型/API 服务：Qwen3.6 / Qwen3.5 系列全部模型，以及指定 embedding 服务：Qwen text-embedding-v4。该模型已经在阿里云百炼平台（https://www.aliyun.com/product/bailian）上开放API调用接口，参赛选手可在百炼平台完成模型接口的调用或者Agent开发。
2. 不允许调用未备案的外部服务。
3. 不允许使用未公开、不可审计的私有数据。
4. 方法应保证可复现，不得依赖赛后无法复现的临时服务或手工流程。

**运行约束：**

1. 对每个数据集，总运行时长（包括运行Agent系统生成预测器、预测器进行预测的时长）不超过 2 小时。
2. 参考硬件为单张 NVIDIA 4090 24GB 或同等规格显卡。
3. 禁止并行运行多个实验进程。
4. 不允许额外人工干预实验过程。
